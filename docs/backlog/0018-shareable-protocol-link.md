## User story

As a user six weeks into a protocol that's working for me, asked by a friend over iMessage "what are you actually doing?" — I want a single button on `#/plan` that produces a shareable URL the friend can tap on their own phone to land inside a redacted, read-only Almanac that shows *my* eat list, *my* avoid list, *my* habit stack, and *my* one-line snapshot, with all marker values, profile data, adherence history, and the API key stripped out — so my friend sees the *real protocol I am holding*, not a sample fixture, without needing to install anything, paste a key, or trust me with their data.

## Why now (four lenses)

### Product Owner

The friend-share story today has two artifacts: the friend-variant printable PDF (ticket 0010, requires AirDrop or print) and the single-marker share card PNG (ticket 0011, one marker at a time). Both are good. Neither is a *link*. The single biggest friction in the current friend-share path is that the friend has to either receive a file or look at a screenshot of a card. There is no way to send a friend a URL that lets the friend interact with the sender's actual protocol the same way the sender does. The sample tour (ticket 0014) gave us the read-only Almanac UI shell — every page works in read-only against a fixture, every write surfaces a tour notice, the consent gate is bypassed safely. The smallest meaningful unit of value here is: a "Share this protocol" button on `#/plan` that base64+lz-encodes a redacted subset of the sender's current `Plan` into the URL hash (`#/shared#<payload>`), and a receiver-side router branch that detects the payload, decodes it, hands it to the existing tour-state shim as if it were a sample fixture, and renders the read-only Almanac against it. No backend, no server, no shortener — the link is literally the data, end-to-end. On the sender side the button copies the link to clipboard and triggers the OS share sheet via `navigator.share`. On the receiver side, the URL hash is the source of truth; refreshing the page re-decodes; closing the tab clears the state.

### Stakeholder

This is the only viral surface a local-first app can ship without weakening the privacy contract. Every other "share with friend" mechanic in the category either (a) requires the friend to sign up, (b) shows the friend nothing more than a screenshot, or (c) requires a backend the friend's data passes through. We have all three rejected on principle. A hash-encoded link is the canonical local-first answer: the data lives in the URL, the URL is the artifact, the recipient's browser is the renderer, no server intermediates. It compounds the **structured artifact** moat — the artifact now travels as a *living document*, not a screenshot — and it compounds the **privacy contract** moat by *demonstrating* the no-backend claim in the most adversarial setting (the friend who has never heard of Almanac and lands cold on a tour-shaped URL with no signup, no key prompt, and no server intermediation visible in the network tab). The hash-payload also has a hard size ceiling (~6KB practical, ~12KB browser-tested) which becomes a useful editorial constraint: we share the *protocol summary*, not the timeline of marker values. The redaction list is the privacy contract made operational: `ownerName` → "A reader's protocol", `anthropicKey` → never serialized, `basedOnPanelIds` → never serialized, `insights` → omitted entirely (they reveal labs by implication), `lifestyle`/`supplements` → included (they're food-adjacent), `eatList`/`avoidList`/`habitStack`/`retest` → included verbatim, `snapshot` → included verbatim.

### User (at 7am on the phone)

Friend texts: "what are you doing for breakfast lately?" I open Almanac, tap Plan, scroll to the bottom where the print buttons live, and now there's a third button: "Share this protocol as a link." I tap it. A tiny editorial confirmation appears: "This will create a link that shows your eat list, avoid list, habit stack, and snapshot — no marker values, no name, no key." I tap "Copy link," paste into iMessage, send. The friend taps the link on her phone. She lands on `#/shared/today` with the same tour-banner from ticket 0014 across the top: "You're reading a friend's shared protocol. Nothing here is yours. Start your own →". She navigates Plan, sees my eat list and habit stack rendered the same way I see them. She taps the avoid list, reads my swaps. She taps "Start your own"; the tour state clears, she lands on `#/welcome` with the consent box. She onboards, pastes her own key, uploads her own labs. The link did the conversion that no screenshot could.

### Growth

This is the single largest acquisition lever the product can ship without a backend, an analytics surface, or a referral code. The friend-PDF (0010) and the share-card PNG (0011) are *artifacts that travel via screenshot or AirDrop*. The shareable link is an *artifact that travels via the same channel the friend already uses to send links* (iMessage, WhatsApp, Telegram, Twitter DM, Reddit comment). The conversion funnel becomes: *visit the friend's shared protocol → see the real artifact (not a sample, not a screenshot) → tap "Start your own" → land on `#/welcome` already convinced*. The 0014 sample tour was the lever for prospects who arrive cold; this ticket is the lever for prospects who arrive *recommended*, which is the higher-converting half of the funnel by a wide margin. Hypothesis: a hash-encoded shareable link converts friend-of-user at 2–4× the rate of a printable PDF, because the friend never has to leave the iMessage thread to read the content; back-of-envelope from comparable patterns (Spotify "shared playlist" links, Cal.com `/demo` routes, Linear's shared issue links). Treat as untested until the next user shares one in the wild. Notably, this ticket is also the *only* surface where a satisfied user can hand-curated-by-them content to a friend without leaking labs — it's the single thing the user is *most likely* to want to share with someone *most likely* to convert, which is the textbook viral-loop shape.

## Acceptance criteria

Each box maps 1:1 to a Playwright test scenario.

- [ ] `src/share/protocol-link.ts` (new) exports `encodeProtocolLink(plan: Plan, snapshot: string): string` and `decodeProtocolLink(hashPayload: string): SharedProtocol | null` where `SharedProtocol` is a redacted subset: `{ snapshot: string; eatList: EatItem[]; avoidList: AvoidItem[]; habitStack: HabitStack; lifestyle: Recommendation[]; supplements: Recommendation[]; retest: RetestItem[]; createdAt: number; }`. The encoder uses `pako` for deflate (already in the package.json? — check first; if not, fall back to a hand-rolled `CompressionStream` since it's available on all evergreen browsers including mobile WebKit). The encoded string is base64url (URL-safe, no padding). No new npm dep if `CompressionStream` is sufficient — confirm in the dev plan.
- [ ] The encoder STRIPS every field not on the SharedProtocol shape. Specifically, the encoder MUST NOT serialize: `Plan.id`, `Plan.generatedAt`, `Plan.basedOnPanelIds`, `Plan.model`, `Plan.insights` (insights reveal marker values by implication — out for v1), `Profile.*` (the encoder takes only `Plan` + `snapshot` as inputs; it cannot see Profile). A typecheck-level safety: the `SharedProtocol` type is the narrowest possible — the encoder's signature is `(plan: Plan, snapshot: string) => string` and the implementation builds the payload from named field copies only (no `...spread`, no `JSON.stringify(plan)`). The test asserts that round-tripping `encode → decode` returns an object whose `Object.keys` are exactly `["snapshot", "eatList", "avoidList", "habitStack", "lifestyle", "supplements", "retest", "createdAt"]` (sorted) and contains no key matching `/name|key|panel|insight|generatedAt|id|model/i`.
- [ ] The decoder returns `null` for any malformed payload (invalid base64, invalid deflate, valid JSON that fails a shape check, payload older than 365 days). The receiver-side router branch (below) treats a `null` decode as an error and routes the visitor to `#/welcome` with a small editorial notice: "This shared protocol link is invalid or no longer readable. Start your own to compose one." Notice is the existing `errorCard()` shape, in the ink token, not oxblood.
- [ ] `src/sample/state.ts` gains a new `enterSharedTour(payload: SharedProtocol): void` and `isSharedTour(): boolean` and `sharedProtocol(): SharedProtocol | null`. The shared-tour mode is a parallel sibling of the sample tour mode (ticket 0014) — both bypass the consent + profile gates, both write nothing to IndexedDB, both render the tour banner. The two modes are mutually exclusive: entering a shared tour calls `exitTour()` first; entering a sample tour calls `exitSharedTour()` first. The shared-tour state lives in memory only — closing the tab clears it; refreshing re-reads the URL hash and re-enters.
- [ ] `src/main.ts` — the router gains a new branch: when the URL hash matches `#/shared#<payload>` (or `#/shared/<route>#<payload>` for sub-routes), the router decodes the payload, calls `enterSharedTour()` with the result, and routes to the inner route (defaulting to `#/today`). When `isSharedTour()` is true, the same gate-bypass branch the sample tour added (ticket 0014) applies — every route renders against the SharedProtocol via the existing tour-state shims, with the Profile and Panels stubbed via a new `sharedTourProfile()` helper that returns the SharedProtocol payload mapped into Profile/Panel shapes where the page renderers expect them.
- [ ] On `#/plan`, alongside the existing "Print or share" controls (ticket 0010), a third button appears: **"Share this protocol as a link."** Tap behavior: (a) open a small inline confirmation strip (ink-token background, ~2 lines tall) reading "This will create a link that shows your eat list, avoid list, habit stack, and snapshot — no marker values, no name, no key." with two buttons "Create link" and "Cancel"; (b) on "Create link," call the encoder, build the URL `${location.origin}/#/shared#${payload}`, copy to clipboard via `navigator.clipboard.writeText`, and call `navigator.share({ url, title: "An Almanac protocol" })` if available (mobile WebKit + Chromium both support `navigator.share` on iOS / Android; fall back silently to "copied to clipboard" on desktop). The confirmation strip then shows a third line: "Link copied. Anyone with this link can read your protocol — there is no way to revoke it short of telling them to delete the message." Editorial, declarative, no exclamation.
- [ ] The shared-tour banner reads: "You are reading a friend's shared protocol. Nothing here is yours. Start your own →". The arrow link routes to `#/welcome`, calls `exitSharedTour()`, and the consent gate is restored. The banner is suppressed for the local user (the sender) — `isSharedTour()` is false when the user is on their own machine viewing their own protocol; the banner only appears when the URL hash payload is the source of state.
- [ ] Write actions in the shared tour are no-ops with the same inline notice the sample tour uses (ticket 0014's `surfaceInlineTourNotice()`): "This is a shared protocol. Start your own to write data." Compose, save check-in, save manual panel, swap meal, generate meals, save profile, delete a panel — every write surfaces the notice and writes nothing.
- [ ] The shared tour bypasses the consent gate but renders the tour banner from the FIRST visit. There is no "consent to view a shared link" step — the link is the consent surface (the friend chose to tap it). The banner is the editorial honest note: this is not yours, no data is stored, nothing here writes.
- [ ] Privacy E2E (`tests/e2e/privacy.spec.ts`) still passes. The shared-tour route never fires an Anthropic call, never widens the egress allow-list. A new traversal asserts the same: visit `#/shared#<payload>`, walk every page in the tour, confirm zero off-allow-list requests.
- [ ] The payload size ceiling: when the encoder produces a string longer than 8192 characters (URL safety for iMessage / most mobile browsers), the sender-side confirmation strip surfaces an error instead of a link: "This protocol is too large to share by link. Use the printable PDF instead." Threshold is documented as 8KB practical; the actual cap may be larger but we are conservative. Realistic protocols (10-item eat list + 8-item avoid + 5 habits + 3 lifestyle + 3 supplements + 2-paragraph snapshot) sit at ~3–5KB compressed; the cap is for users with unusually long custom plans.
- [ ] The shared-tour render is byte-identical to a normal tour render for the included fields. The visual: same `#/today` layout, same Plan-page sections, same Meals-page layout (the Meals page renders an editorial empty state — "This shared protocol does not include the meal plan"), same Progress-page editorial empty state — "This shared protocol does not include marker data."), same Compare and Recap pages omitted entirely (no marker data, no check-ins to recap). The Today screen omits the streak strip and the check-in panel — both require local check-ins that the shared payload doesn't carry — and renders the eat list + habit stack + snapshot only.
- [ ] Renders on both **chromium and mobile-webkit**. Mobile is in-scope — the share-this-link-with-a-friend channel is mobile-dominant. The `navigator.share` happy path is asserted on mobile-webkit; the clipboard fallback is asserted on chromium.
- [ ] New `tests/e2e/shared-protocol.spec.ts` covers: encoder round-trip (every redaction holds), decoder rejects malformed payloads, `#/shared#<payload>` routes into the shared tour and bypasses gates, every page in the shared tour renders, write actions surface the no-op notice, "Start your own" clears the shared tour and restores the consent gate, size-cap error fires when payload exceeds 8KB, shared-tour banner appears only when payload-driven state is active, zero Anthropic calls across a full shared-tour traversal.
- [ ] `tests/e2e/privacy.spec.ts` — extend with a traversal of `#/shared#<known-payload>` across every shared-tour route and confirm no off-allow-list egress.

## Out of scope

Explicit anti-goals. The dev agent will not do these even if they seem related.

- A backend-hosted short-link service. The whole point is that the data IS the link; an intermediating server would break the local-first contract. Long URLs sent via iMessage / WhatsApp / Telegram are the acceptable cost of the privacy guarantee.
- A "revoke this link" mechanism. We cannot revoke a link whose payload is its data; the only "revocation" is telling the recipient to delete the message. The confirmation strip says this verbatim — the user consents to that property at share time, not at link-creation time.
- Including marker values, panel data, or `Plan.insights[]` in the shared payload. Insights reveal lab data by implication ("ferritin 32 ng/mL") and would defeat the redaction discipline. A future ticket could add an optional "include the snapshot's editorial summary of what stood out, without numbers" mode if user feedback asks for it; not for v1.
- A "compose a fresh plan from this shared protocol's eat list" import button on the receiver side. The friend's onboarding flow stays the same; the shared tour is read-only, full stop. Mixing the two would muddy the local-first contract.
- An email-the-link button or a QR code generator. The OS share sheet (`navigator.share`) is the only sender-side surface. Email and QR can come later if asked for; they're not on the critical path.
- Encrypting the payload with a recipient-side passphrase. The hash payload is plaintext (post-deflate) — anyone with the link can read it. Encryption would require a key-exchange step that doesn't exist in a no-backend product.
- A "link analytics" pane in Settings. No analytics, no telemetry, no "how many people opened your link" — the user is the only counter and they don't get one.
- Versioning the payload schema with a separate field. The payload's first byte is the format version (a single ASCII char); the decoder validates it; older payloads stop working after one breaking change. The "older than 365 days" guard above is operational, not version-driven.
- An open-graph image preview when the link is shared. We do not run a server, so we cannot serve dynamic `og:image` content; static image previews in the Vercel deployment are fine if they exist, but generating a per-link preview is out.

## Engineering notes

Files / patterns the dev should touch. Be specific enough that the dev doesn't have to re-discover the architecture.

- `src/share/protocol-link.ts` (new) — `encodeProtocolLink(plan, snapshot)` + `decodeProtocolLink(hashPayload)`. The encoder: (1) build the `SharedProtocol` payload from named field copies (no spread); (2) JSON-stringify; (3) UTF-8 encode; (4) deflate via `CompressionStream("deflate")` — supported on Chromium, mobile-WebKit (iOS 16+), and modern Firefox; no new npm dep needed; (5) base64url-encode the bytes. The decoder reverses the pipeline with `DecompressionStream("deflate")`. Format version: prepend a single `v1` prefix to the JSON before deflate; decoder reads the prefix first and rejects unknown versions. The size guard runs on the base64url string and caps at 8192 chars.
- `src/sample/state.ts` — add `enterSharedTour(payload)` / `exitSharedTour()` / `isSharedTour()` / `sharedProtocol()`. The shared-tour mode lives alongside the sample-tour mode; both are read by the existing `isTour()` umbrella check (rename to `isAnyTour()` if needed for clarity, or keep `isTour()` returning `true` for either mode and add a separate `isSharedTour()` for the banner copy distinction). The shared-tour payload lives in module-level state (`let cached: SharedProtocol | null = null`) — no localStorage; the URL hash is the durable source of state. Refreshing the page re-runs the router branch and re-decodes from the URL.
- `src/db.ts` — extend the existing tour-aware read shims (from ticket 0014) to consult `isSharedTour()` as well. `getProfile()` returns a synthesized profile in shared-tour mode with `ownerName: "A reader"`, every sensitive field stubbed. `allPanels()` returns an empty array (the Progress page already has a no-panels editorial branch). `latestPlan()` synthesizes a Plan from the SharedProtocol payload — the existing Plan-page renderer reads it; missing fields (`insights`, `basedOnPanelIds`, etc.) render as their existing empty branches. `latestMealPlan()` returns `undefined`. `recentCheckIns()` returns `[]`. The write guards from 0014 already cover the shared-tour case if `isAnyTour()` is the gate.
- `src/main.ts` — the router gains a `#/shared` branch. Match pattern: `/^#\/shared(?:\/([^#]*))?#(.+)$/`. The first capture is the inner route (default "today"); the second capture is the payload. The branch: decode → on success, `enterSharedTour(payload)` + route to `#/${inner}`; on failure, route to `#/welcome` with an error query (`#/welcome?invalid-share=1`) the welcome page reads to surface the editorial notice.
- `src/pages/welcome.ts` — when `?invalid-share=1` is on the URL, render a slim editorial notice above the consent box: "A shared protocol link couldn't be read — it may be malformed or out of date. Start your own to compose one." The notice uses the existing `tourNotice()` template from 0014.
- `src/pages/plan.ts` — add the third button alongside the existing "Print or share" controls. The confirmation strip uses the existing inline-confirmation pattern from the print panel (search for `print-confirm` or whichever class the print-confirm uses). On Create-link: encoder → clipboard → `navigator.share` (try/catch — `navigator.share` rejects on user-cancel, which is fine to swallow). On Cancel: hide the strip.
- `src/chrome.ts` — `masthead()` already handles `isTour()` → render banner. Branch on `isSharedTour()` for the new banner copy: "You are reading a friend's shared protocol. Nothing here is yours. Start your own →". The arrow link routes to `#/welcome` and calls both `exitTour()` and `exitSharedTour()`.
- `src/ui.ts` — `surfaceInlineTourNotice()` (from 0014) already exists; extend the notice copy to read "This is a shared protocol. Start your own to write data." when `isSharedTour()` is true, falling back to the existing sample-tour copy otherwise.
- `src/styles.css` — re-use `.tour-banner` from 0014; no new color tokens. Add `.share-link-confirm` (the inline confirmation strip on `#/plan`) — same shape as the existing print-confirm strip.
- `tests/helpers/flows.ts` — add `enterSharedTour(page, payload)` and `exitSharedTour(page)` helpers. The first wraps `page.goto("/#/shared#" + payload)`; the second clicks the banner's "Start your own" link.
- `tests/e2e/shared-protocol.spec.ts` (new) — every acceptance bullet maps to a `test()`. Re-use the existing `composePlan` flow to get a real Plan in IndexedDB, then drive the Plan page's share-link button, capture the link from the clipboard (use Playwright's clipboard API), assert the payload round-trips, then `page.goto` the link in a fresh browser context to assert the receiver-side behavior.
- `tests/e2e/privacy.spec.ts` — extend the existing tour-traversal allow-list assertion to include a shared-tour traversal of every page that renders in shared-tour mode (Today, Plan, Meals empty, Progress empty, Welcome). Same allow-list; new traversal.
- Schema migration: **no**. The shared tour writes nothing.
- Egress allow-list change: **no**.
- New deps: **no** if `CompressionStream` is sufficient (it is on every target browser). If the dev discovers an edge case, the fallback is `pako` (~45KB minified) — but the spec assumes no new dep and the implementation should land that way.
- Voice spec for the confirmation strip and the banner: editorial, declarative, no exclamation, no "amazing/journey/exciting". Samples in the acceptance bullets are the calibration bar.
