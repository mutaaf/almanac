## User story

As a technically-literate user reading my Plan after a fresh draw, I want to tap any marker on my latest panel, drag its value along the functional-range thermometer, and see in real time which insights from the deterministic rule engine would appear, disappear, or change priority — and which habits would get added or dropped — so that the rule engine becomes a visibly *interactive* artifact I can interrogate, instead of a black box that emits a finished protocol; so that when a clinician asks "what if her hemoglobin had been 13.5?" I can demonstrate the answer on the spot rather than wait for a follow-up draw.

## Why now (four lenses)

### Product Owner

The deterministic insight engine (`src/insights.ts`) is the structural advantage no LLM wrapper can copy: a curated, multi-marker, threshold-aware ruleset that runs in pure TypeScript and produces the authoritative findings before any Claude call. Ticket 0013 made the engine's output *auditable* (the "Why this fired" slideover names the rule, the supporting markers, the thresholds). Ticket 0016 carried that auditability onto paper for the clinical channel. Both surfaces show the engine's output at a single point in time — they don't show what the engine *would* output if a value were different. The smallest meaningful unit of value is one in-place interaction on `#/plan`: tap any marker chip in the "supporting markers" or panel summary, drag a slider that overlays the existing `thermometer()` viz, and the page's insights list re-renders live with three categorical badges per insight — "still fires," "would no longer fire," "would newly fire." No Anthropic call. No new schema. The rule engine is already pure; we are just calling it twice (current panel, hypothetical panel) and diffing the results. The engine's purity is the lever.

### Stakeholder

This is the most direct widening of the **programmatic insight engine** moat we have left to ship. The engine has been: (1) load-bearing for the prompt since v1, (2) auditable in-app since 0013, (3) auditable on paper since 0016. The natural next state is *interactive* — a user can manipulate inputs and watch outputs change, and the manipulation is a property only local-first architecture can ship credibly. A backend-shaped product would have to route every slider drag through an API call, which is slow, traceable, and structurally rate-limited; Almanac's engine runs in 1–5ms per evaluation in pure JavaScript on the user's device, so the slider produces 60fps reactivity for free. This is the kind of capability that defines what an editorial precision-health product looks like in 2026: the *evidence* behind every recommendation is not just visible, it is *manipulable*. It also opens the wedge for clinical channels — a physician in a 20-minute appointment can ask "what would change if this marker were in range?" and see the answer rendered in the editorial almanac voice, not in a clinic-portal table or a chat reply. The interaction is the proof of the engine's seriousness in a way that prose claims cannot match.

### User (at 7am on the phone)

I'm on `#/plan` after a fresh draw. My ferritin sits at 32 ng/mL on the thermometer. Below the eat list, I see the insight: "iron-restricted erythropoiesis pattern · ferritin 32 ng/mL · MCV 86 fL." I tap the small "What if?" chip on the right side of the ferritin row. The thermometer becomes interactive — a small handle appears at the current value, I can drag it left or right within the lab range. As I drag toward 60 ng/mL, two things happen in the insights list above: (1) "iron-restricted erythropoiesis" gets a strikethrough and a small "would no longer fire" badge, (2) a fresh insight slides in below with a "would newly fire" badge — "ferritin sufficiency: holding above the functional floor; iron tier can move from advanced to easy." I release the drag. The chip footer reads "You moved ferritin from 32 to 60 ng/mL. Three insights changed." I tap "Reset to current values." Everything snaps back to the live panel. I tap "Keep my real value." The simulator closes. The Plan reads as it did before I started.

### Growth

This is the moat-deepening ticket of the three, not an acquisition ticket — the audience is the user already inside the app. But it produces the highest-density "show me" artifact we have access to: a short screen-recording of a marker value being dragged while the insights list reorganizes in real time, in the editorial almanac voice. That is the artifact a serious precision-health forum will share — the same way Excalidraw's "infinite canvas" demo or Linear's command-palette video do organic acquisition through pure technical credibility. (Hypothesis: a 6-second screen recording of the what-if simulator on a credible marker — ApoB, ferritin, or fasting insulin — generates ~3-5× the impression-to-click rate on a precision-health forum compared to a static screenshot, because the *interactivity* is the claim; back-of-envelope from comparable demo-video mechanics on r/PeterAttia and r/QuantifiedSelf — treat as untested.) It also compounds with every future rule we add to `src/insights.ts`: every new entry in `RULES[]` or `TREND_RULES[]` automatically becomes interrogable through the simulator with no per-rule UI work. The discipline 0013 enforced (every rule has a `RULE_GLOSSES` entry, every supporting-marker chain is typed) pays out a third time here.

## Acceptance criteria

Each box maps 1:1 to a Playwright test scenario.

- [ ] `src/insights/simulate.ts` (new) exports `simulateInsights(panel: Panel, override: { markerKey: string; value: number }): { current: PreComputedInsight[]; hypothetical: PreComputedInsight[]; diff: SimulationDiff }`. The function clones the panel, replaces the named marker's value, re-runs the same `computeInsights()` path used in production (no parallel rule engine — one source of truth), and computes the diff. `SimulationDiff` shape: `{ stillFires: string[]; wouldNoLongerFire: string[]; wouldNewlyFire: string[] }` — three arrays of rule ids.
- [ ] The function is **pure**: no DOM, no Dexie, no network. Asserted by importing and calling it from the spec context (`page.evaluate`) with a synthetic panel and confirming the diff against an expected snapshot. The simulator never persists; the live panel in IndexedDB is untouched.
- [ ] On `#/plan`, every marker visible on the page gains a small "What if?" chip next to its value (the same row pattern the provenance "Why this fired" chip uses from 0013). The chip is keyboard-focusable, has `role="button"`, and is styled as a small-caps eyebrow link in the ink token (not oxblood — interrogation is informational, not a call-to-action). The chip appears on (a) markers in the supporting-markers row of any insight card, (b) markers in the panel summary block. The same marker appearing in both places shows the same chip rendered identically.
- [ ] Tapping the chip enters a per-page simulation mode for that marker. The marker's thermometer viz gains an overlay slider (rendered via existing `thermometer()` in `src/viz.ts` — add an optional `interactive?: { onChange: (value: number) => void; bounds: [number, number] }` parameter). The slider bounds are `[max(0, current * 0.3), current * 2.5]` for most markers, or the lab range from `RESULT.range` when present (lab range wins when it provides tighter bounds). The slider snaps to one decimal place for percentage-style markers, integers for count-style markers, and the marker's natural precision otherwise (resolve from `MarkerDef.precision` if defined; default to two significant figures).
- [ ] While in simulation mode for a marker, every insight card on the page re-renders within 50ms of a slider input event, with one of three states added to the card's eyebrow: `still fires` (no styling change, no badge), `would no longer fire` (the card's title gets a strikethrough, the card body fades to 60% opacity, a small "would no longer fire" badge in the ink token replaces the priority tag), `would newly fire` (a new card slides in at the bottom of the insights list with a small "would newly fire" badge and the rule's `evidence` string + `RULE_GLOSSES[ruleId]` paragraph — re-used from 0013, no new content authored).
- [ ] A persistent simulation header renders at the top of the Plan page when simulation mode is active: "Simulating <marker>: <currentValue> → <draggedValue> <unit>. <N> insights changed." The header has two buttons: "Reset to current" (snaps the slider back to the panel value, leaves simulation mode active) and "Keep my real value" (exits simulation mode entirely, restores the original page render). Closing simulation mode never writes anything — the marker on the live panel is unchanged.
- [ ] When simulation mode is active, the eat list, avoid list, habit stack, and meal plan summary on the Plan page do NOT re-render. Only the insights list is reactive. (The rule engine emits insights, not eat-list deltas; surfacing a hypothetical eat list would require an LLM call, which the ticket explicitly avoids.) The eat list / habit stack get a small note above them: "These are based on your real values. Close the simulation to see how they would shift."
- [ ] Simulation runs with **zero Anthropic calls**. Asserted across an extended drag (50 input events) via the mock's request count remaining at zero.
- [ ] Multi-marker simulation: a user can enter simulation mode for one marker, then tap "What if?" on a second marker while the first is still active. The second marker's slider stacks under the header. Both overrides are applied to the same `simulateInsights()` call (the function accepts `Array<{ markerKey: string; value: number }>` — the single-override case is a degenerate array). The header updates: "Simulating ferritin: 32 → 60 ng/mL and MCV: 86 → 91 fL. Four insights changed." Closing one override removes only that override; closing all returns to live.
- [ ] User-defined markers (from 0002) work in the simulator with the same interaction. The bounds fall back to the user-defined range; if no range is defined, the slider bounds use a wider `[current * 0.5, current * 2]` window and the chip footer notes "This is a user-defined marker — bounds are approximate."
- [ ] Trend rules from `TREND_RULES[]` are simulated by treating the override as a hypothetical *new* draw appended to the time series. The diff for trend rules reads "would no longer fire after the next draw" / "would newly fire after the next draw" — the temporal hedge is in the badge copy. Tested with a fixture that has three historical ferritin draws plus a current value the user is dragging.
- [ ] Provenance (from 0013) for "would newly fire" cards is rendered the same way it is for real cards — the "Why this fired" chip is present, the slideover opens, the supporting-marker rows reflect the *hypothetical* value of the dragged marker (the spec asserts the slideover's value row reads the dragged number, not the live one).
- [ ] The simulator does NOT add a "would no longer fire" or "would newly fire" entry to any persisted `Plan.insights[]` array. The diff is render-time only. Asserted by closing the simulator and confirming `db.latestPlan()` is byte-identical to the snapshot taken before simulation began.
- [ ] Renders on both **chromium and mobile-webkit**. On mobile-webkit, the slider becomes a tap-to-edit number input plus +/- buttons (a true touch-drag slider on the thermometer overlay is flake-prone on mobile-webkit — the same precision is achieved through a stepped input). The simulation header and the insight badge logic are identical across both projects.
- [ ] Privacy E2E (`tests/e2e/privacy.spec.ts`) still passes — no new hostnames; the simulator fires no network requests; no telemetry of slider events.
- [ ] Schema migration: **none**. The simulator reads the live panel, computes locally, and renders; nothing is persisted.
- [ ] New `tests/e2e/what-if.spec.ts` covers: chip appears on every marker on the Plan page, single-marker drag re-renders insights, "would newly fire" badge surfaces a rule that wasn't firing, "would no longer fire" badge strikes through a card that was firing, multi-marker simulation stacks correctly, trend rules surface the "after the next draw" hedge, user-defined marker bounds fall back correctly, simulator close restores original page state, zero-network assertion across 50 drag events, `db.latestPlan()` is unchanged after simulation, provenance slideover reflects the hypothetical value during simulation.

## Out of scope

Explicit anti-goals. The dev agent will not do these even if they seem related.

- An LLM call to regenerate the eat list / avoid list / habit stack for the hypothetical state. The whole point of the simulator is that it shows what the *deterministic* engine outputs change to. The eat list is downstream of the LLM compose call; simulating it would require a second compose, which is slow, expensive, and structurally unfair to interrogate (the LLM's output is non-deterministic). The note above the eat list ("Close the simulation to see how they would shift") is the honest framing.
- Saving a simulation as a "what-if plan." The simulator is read-only and ephemeral. Persisting hypothetical plans would muddy the persistent timeline and create a new class of "is this real or simulated?" footgun.
- A "share this simulation" PNG. The marker hero share card (0011) covers single-marker claims; the simulator is an interactive surface, and a static PNG of a simulated state would be misleading without the slider context. The screen-recording artifact described in the growth lens is something users produce themselves through the OS.
- A multi-rule "what if all my markers were in range" preview. The simulator is per-marker (or per-small-set); a "fix everything at once" button would defeat the educational purpose, which is to see *which specific marker* drives *which specific insight*.
- A "compare two hypothetical states" view. Single-axis simulation is the discipline; a two-state diff is a different surface (the existing 0009 side-by-side compare already covers two real states).
- A confidence interval on the dragged value. The slider produces a single hypothetical value; the rule engine is threshold-based, so the answer is binary at each value. No statistical hedging.
- Editing the rules from the simulator. The rules live in TypeScript by design (per 0013); the simulator interrogates them, it does not author them.
- Simulating on `#/progress` against historical draws. The simulator lives on `#/plan` because that is where the deterministic insights are surfaced. Adding simulation to Progress would double the surface area without doubling the user value.
- An LLM-narrated explanation of why each diff happened ("because ferritin crossed the female functional floor, the iron-restricted pattern stopped firing"). The provenance slideover from 0013 already shows the rule's machinery; the diff badges name the change tersely. Narrating the diff is editorial dead weight when the user can read the rule's gloss directly.

## Engineering notes

Files / patterns the dev should touch. Be specific enough that the dev doesn't have to re-discover the architecture.

- `src/insights/simulate.ts` (new) — `simulateInsights(panel, overrides)`. The function clones the panel via structured-clone (deep, since `panel.results` is an array of objects), applies each override by mutating the cloned result's `value` field, and calls the existing `computeInsights()` export from `src/insights.ts`. The diff is computed by `id`-keyed set operations on the two returned arrays. The signature accepts `overrides` as an array so the multi-marker case is a degenerate `[single]`.
- `src/insights.ts` — `computeInsights()` is already pure (it takes a `Panel`-shaped input and a context resolver and returns `PreComputedInsight[]`). No changes here unless the function's internal state caching needs an explicit "this is a simulation, skip the cache" flag — the spec asserts no cache contamination between calls.
- `src/viz.ts` — `thermometer()` gains an optional `interactive?: { value: number; bounds: [number, number]; onChange: (v: number) => void }` parameter. When set, the rendered SVG includes a draggable handle with the standard pointer-events wiring; when omitted, the existing render is unchanged. The slider snaps to the marker's precision (resolve from `MarkerDef.precision` — add a per-marker `precision?: number` field to `MarkerDef` with curated defaults for the markers that matter: ferritin (integer), ApoB (integer), hsCRP (one decimal), fasting insulin (one decimal), hba1c (one decimal), vit-D (integer), MCV (integer), magnesium-RBC (one decimal). Other markers default to two significant figures via a small utility.
- `src/data/markers.ts` — add the `precision` field to the curated marker entries listed above. Additive, no migration.
- `src/types.ts` — extend `MarkerDef` with optional `precision?: number`. Add `SimulationDiff` and `SimulationOverride` shapes. All additive.
- `src/pages/plan.ts` — render the "What if?" chip next to every marker value on the page. Wire chip click to a new page-local state machine (`simulationState: { overrides: SimulationOverride[] } | null`). When non-null, the chip's marker enters interactive mode, the simulation header renders at the top of the page, and the insights list passes through a diff-aware renderer that reads from `simulateInsights(livePanel, overrides)`. On close, the page re-renders against the unmutated live state. The state machine is per-render, not persisted — closing the page or navigating away exits simulation mode.
- `src/styles.css` — `.sim-header` (full-width banner styled like the tour banner from 0014 — restraint, ink token, hairline rule), `.sim-chip` (mirrors `.insight__provenance-chip` from 0013), `.insight--would-no-longer-fire` (strikethrough on the title, 60% opacity on the body), `.insight--would-newly-fire` (small "would newly fire" badge in ink, no other color change), `.thermometer__slider-handle` (the draggable handle on the SVG thermometer). No new colors.
- `src/insights.ts` — `RULE_GLOSSES` is the source of truth for the "would newly fire" card's body content. No new gloss entries are authored by this ticket; the discipline from 0013 (every rule has a gloss) is the prerequisite.
- `tests/helpers/flows.ts` — add `enterSimulation(page, markerKey)` and `dragSimulationSlider(page, markerKey, value)` helpers. The drag helper triggers the slider's `input` event programmatically (mobile-webkit) or the pointer-drag (chromium) depending on the project — `page.context().browserName` branch covers both.
- `tests/e2e/what-if.spec.ts` (new) — every acceptance bullet maps to a `test()` or `expect()`. The "byte-identical live plan" assertion serializes `db.latestPlan()` before and after, asserts `JSON.stringify(before) === JSON.stringify(after)`. The 50-drag zero-network assertion drives the slider through 50 input events and asserts `mocks.requestCount("api.anthropic.com")` is 0 after.
- Schema migration: **no**.
- Egress allow-list change: **no**.
- New deps: **no**.
- Voice spec for the simulator copy: the header reads "Simulating <marker>: <current> → <hypothetical> <unit>." The badges read "would no longer fire" and "would newly fire" — declarative, conditional, no "" / "no longer "; the hedge is in the tense. The Reset and Close buttons are labelled "Reset to current" and "Keep my real value." No exclamation, no "amazing/journey/exciting." The note above the eat list reads "These are based on your real values. Close the simulation to see how they would shift." — declarative, honest about the LLM-downstream limitation.
